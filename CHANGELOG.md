# 更新记录

本文档记录项目从 MVP 到后续迭代的主要功能演进，面向后续维护、README 同步更新和阶段性复盘使用。

记录口径：

- 本文不是逐次保存文件或逐次 Codex 对话级别的审计日志。Git 只完整记录已经提交的代码快照和每次提交的 diff，不记录未提交前的所有中间编辑过程。
- 早期记录优先依据 Git 证据重建：提交时间、提交 hash、`git show --stat` 的文件变更范围、关键代码 diff，以及当前代码中的接口、数据库表和前端路由。
- 提交信息本身可能不够规范，因此本文不照抄 commit message，而是把实际代码变化整理成专业的产品/工程变更描述。
- 后续更新请把新内容追加到顶部的“未发布”区块；发布或阶段完成后，再归档到对应日期。
- 同一天内按更新时间倒序排列。没有明确提交或发布时间的内容，应写入“未发布”，并说明依据来自工作区 diff、截图、手动记录或开发记忆。

维护建议：

- 查看某次提交的证据：`git show --stat <commit>` 和 `git show <commit>`。
- 查看两次提交之间的功能变化：`git diff <old_commit>..<new_commit>`。
- 查看当前尚未提交的文档或代码变化：`git status --short` 和 `git diff`。
- 写记录时优先描述用户可感知的功能、接口/数据模型变化、配置变化和维护风险；不要只描述“改了某个文件”。
